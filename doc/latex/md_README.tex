7Guard uses a encryption tecnique called \href{https://en.wikipedia.org/wiki/One-time_pad}{\texttt{ O\+TP}} (One-\/\+Time-\/\+Pad) which sums the bytes from the file with a random number, saving the sum to the crypted file and the random number will be used as the key file.


\begin{DoxyCode}{0}
\DoxyCodeLine{file\_to\_crypt + random\_number = crypted\_file}
\DoxyCodeLine{                      |               |}
\DoxyCodeLine{                      |                -\/-\/>Stored in crypted file (*.7cy)}
\DoxyCodeLine{                       -\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/-\/>Stored in key file (*.7ky)}
\end{DoxyCode}


To decrypt it\textquotesingle{}s just inverting the process.


\begin{DoxyCode}{0}
\DoxyCodeLine{original\_file = crypted\_file -\/ random\_number}
\end{DoxyCode}


The issues with this tecnique are\+:
\begin{DoxyItemize}
\item Generation of truly random numbers
\begin{DoxyItemize}
\item The problem was solved by using the Intel functions to generate random numbers in hardware level (more info \href{https://en.wikipedia.org/wiki/RdRand}{\texttt{ here}}).
\end{DoxyItemize}
\item The crypted and key file will have the same size of the original one.
\begin{DoxyItemize}
\item In this version you need to deal with it. I\textquotesingle{}m already working on a implemetation with a server (more for enterprise use) where the keys will be stored on the enterprise server.
\end{DoxyItemize}
\end{DoxyItemize}\hypertarget{md_README_autotoc_md21}{}\doxysection{Usage}\label{md_README_autotoc_md21}
